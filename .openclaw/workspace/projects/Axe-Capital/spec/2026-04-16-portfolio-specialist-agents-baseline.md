# Portfolio Specialist Agents Baseline Spec

**Date:** 2026-04-16
**Status:** Approved for implementation
**Scope:** Specialist analyst agents for the IBKR portfolio baseline and monthly reruns

---

## Goal

Build the first usable specialist-agent workflow for Axe Capital:

1. Read the current portfolio from Interactive Brokers through the existing Step 5 portfolio artifact.
2. Run Fundamental, Technical, and Macro analysis for every current portfolio symbol, including ETFs.
3. Save structured reports that the dashboard and Step 3 decision pipeline can consume.
4. Let Step 3 decisions use the specialist reports instead of running from generic portfolio/news context only.

This is not a watchlist workflow. The portfolio source of truth is IBKR only.

---

## User Workflow

After opening the PC, the user starts TWS or IB Gateway, logs into IBKR, and leaves it running. The app runtime is launched from the project with:

```bash
cd ~/.openclaw/workspace/projects/Axe-Capital
./step7-automation/start_prototype.sh
```

The dashboard opens at:

```text
http://localhost:5173
```

For the first specialist baseline, the user runs:

```bash
cd ~/.openclaw/workspace/projects/Axe-Capital/step7-automation
python -m axe_orchestrator.cli run specialists
```

For the monthly rerun, the same command is scheduled by cron or run manually. The workflow refreshes the portfolio first, then analyzes the symbols in `step6-dashboard/public/portfolio.json`.

---

## Data Source Rules

- Portfolio membership comes only from `step6-dashboard/public/portfolio.json`.
- That file is generated by Step 5 from IBKR when `AXE_PORTFOLIO_SOURCE=ibkr` or `auto`.
- `targets.json` is not used for portfolio baseline membership.
- Symbols with zero or negative position are ignored.
- ETFs are included. They receive the same three specialist report types, but the written analysis must acknowledge ETF/sector/index exposure when source data indicates it.

---

## Outputs

Each specialist report writes a timestamped JSON file and Markdown file under:

```text
step6-dashboard/public/analyst-reports/
```

File pattern:

```text
<SYMBOL>-<agent>-<YYYY-MM-DDTHH-MM-SSZ>.json
<SYMBOL>-<agent>-<YYYY-MM-DDTHH-MM-SSZ>.md
```

Agent names:

- `fundamental`
- `technical`
- `macro`

The report runner also writes:

```text
step6-dashboard/public/analyst-reports/index.json
```

The index maps each symbol to the latest report path for each agent, so Step 3 and the dashboard do not need to guess which timestamped file is newest.

---

## Report JSON Contract

All three agents return the same top-level shape:

```json
{
  "generated_at": "2026-04-16T00:00:00Z",
  "ticker": "GOOG",
  "agent": "technical",
  "summary": "Concise thesis.",
  "key_findings": ["Finding one", "Finding two"],
  "data_sources": ["yfinance"],
  "confidence": 6,
  "report": "Human-readable detailed report text.",
  "metrics": {
    "agent_specific_key": "agent_specific_value"
  }
}
```

Required behavior:

- `ticker` is uppercase.
- `confidence` is an integer from 1 to 10.
- `summary`, `key_findings`, and `report` must be present even when market data is incomplete.
- `data_sources` must name the source families used.
- `metrics` carries agent-specific structured values for later UI work.

## Agents

### Fundamental Analyst

Existing package: `step8-fundamental`

Responsibilities:

- Business quality
- Revenue, profitability, margins, valuation indicators
- Major data gaps and thesis risks

Data:

- `yfinance.Ticker(...).info`

Output agent id:

- `fundamental`

### Technical Analyst

New package: `step9-technical`

Responsibilities:

- Trend and regime
- Moving averages
- Recent high/low range
- Support, resistance, and invalidation framing

Data:

- `yfinance.Ticker(...).history(period="1y", interval="1d")`

Output agent id:

- `technical`

### Macro Strategist

New package: `step10-macro`

Responsibilities:

- Sector and asset-class context
- Rate, liquidity, USD, and market-regime framing where available
- How macro conditions can help or hurt the specific holding

Data:

- `yfinance.Ticker(...).info` for sector/category context
- ETF/market proxy history from yfinance when available

Output agent id:

- `macro`

## Orchestration

Step 7 adds a portfolio specialist runner:

```bash
python -m axe_orchestrator.cli run specialists
```

Behavior:

1. Run Step 5 portfolio refresh first.
2. Read symbols from `step6-dashboard/public/portfolio.json`.
3. For each symbol, run Fundamental, Technical, and Macro.
4. Continue after a single symbol/agent failure and return a non-zero exit code if any report fails.
5. Regenerate `health.json`.

Step 7 also supports targeted specialist runs:

```bash
python -m axe_orchestrator.cli run fundamental GOOG
python -m axe_orchestrator.cli run technical GOOG
python -m axe_orchestrator.cli run macro GOOG
```

## Decision Integration

Step 3 adds analyst report context to the decision payload:

```json
{
  "analyst_reports": {
    "fundamental": { "...": "latest fundamental report for ticker" },
    "technical": { "...": "latest technical report for ticker" },
    "macro": { "...": "latest macro report for ticker" }
  }
}
```

The Bull, Bear, CRO, and CEO prompts must know that specialist reports are available in the context and should cite data gaps rather than inventing facts.

Step 7 removes the hardcoded `MSFT` behavior by allowing:

```bash
python -m axe_orchestrator.cli run decision GOOG
```

If the dashboard calls `/refresh/decision` without a ticker, Step 7 may still use a safe default for compatibility, but CLI decisions must accept an explicit ticker.

## Acceptance Criteria

- `step9-technical` exists and can write JSON + Markdown reports.
- `step10-macro` exists and can write JSON + Markdown reports.
- `step8-fundamental` updates `analyst-reports/index.json` when it writes a report.
- `axe run specialists` runs reports for every positive-position symbol in `portfolio.json`.
- `axe run decision <TICKER>` passes the requested ticker to Step 3.
- Step 3 loads latest analyst reports for the ticker from `analyst-reports/index.json`.
- Unit tests cover symbol extraction, report index updates, targeted runner arguments, and Step 3 report loading.
- The dashboard remains accessible at `http://localhost:5173` after starting `./step7-automation/start_prototype.sh`.
